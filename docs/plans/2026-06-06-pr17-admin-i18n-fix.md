# Fix pre-merge PR #17 — labels admin hardcodeados (i18n)

> Brief de Cowork → Claude Code CLI. Trabajar **sobre la branch `feature/feedback-i18n-finish`** (NO sobre `main`). Al terminar, gate verde y se mergea PR #17.

## Contexto

Review de PR #17 pasó 5/6 checks. Único hallazgo: las admin sub-pages aplicaron el patrón `SubmitButton` pero dejaron **~25 labels hardcodeados** en `children` / `pendingLabel` / `confirmMessage`, con idioma mezclado (mayoría ES, `settings` en EN). El storefront quedó 100% traducido. Hay que pasar esos literales por `t(locale, …)` reusando el namespace `admin.action.*` que **ya existe** en `lib/i18n/messages.ts`.

`MAINTENANCE_MODE` no se toca.

## Convención

- Keys nuevas van en `admin.action.*` (acción) y reusan `common.pending` para el `pendingLabel` genérico `"…"`.
- Toda key nueva se agrega en **tres** lugares de `lib/i18n/messages.ts`: (1) el `MessageKey` union type, (2) bloque `en-US`, (3) bloque `es-419`. El test de paridad (`lib/i18n/__tests__/messages.test.ts`) falla si EN y ES no tienen exactamente las mismas keys.
- `confirmMessage` también debe ir por `t()` (no literal).
- Labels dinámicos que ya interpolan (`→ {s}` en orders, `{p.isActive ? … : …}`) se traducen los textos, no el arrow.
- **`SubmitButton` recibe `children: React.ReactNode`** (no function-children): los textos se resuelven en el server component padre con `t(locale, key)` y se pasan ya resueltos. `locale` ya está en scope en estas páginas (son async server components que ya llaman `t(locale, …)` en otras partes del mismo archivo). No hace falta tocar la firma del componente.
